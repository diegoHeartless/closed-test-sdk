# Roadmap: closed-test SDK (Android)

Дорожная карта привязана к обсуждению концепции **мониторинга закрытого тестирования** (не маркетплейс тестеров): тестеров ищут в Telegram/Reddit и т.п., продукт даёт **прозрачность и доказательство активности** через SDK в приложении разработчика.

---

## Продуктовое саммари (зафиксировано)

| Что это | Что это не |
|--------|------------|
| Контроль процесса closed testing под ожидания Google Play | Поиск/продажа тестеров внутри продукта |
| Дашборд: дни, активность, кто «выпадает», напоминания | «Гарантия прохождения модерации» и формулировки про накрутку |
| Опциональное приложение тестера: join по ссылке, прогресс, пуши | Обязательный тестерский клиент для работы трекинга |

**Ключевая ценность:** после интеграции SDK процесс виден **даже если тестеры не ставят ваше приложение** — данные идут из тестируемого APK. Приложение тестера упрощает join и уведомления.

**Философия из обсуждения:** без жёстких санкций и «токсичного минус-рейтинга» — в основном **сигналы качества** (валидные сессии, метрики, прозрачность). Разработчик сам отбирает людей; SDK и бэкенд отвечают за **честность метрик**, а не за моральный суд.

**Сайт (отдельный слой):** не каталог-поиск, а **структурированный пост теста** + ссылка в приложение + профиль с позитивными метриками (SEO и доверие).

---

## Цели именно этого репозитория (SDK)

- Минимальная интеграция в Android-приложение: **одна инициализация + автоматические сессии**, опционально явные события.
- Отправка событий на ваш бэкенд по **API key / test id** из кабинета.
- Поддержка **привязки тестера** к потоку событий (см. ниже).
- Прозрачность для разработчиков: желательно **open-source клиент** (см. раздел «Доверие»).

Платформа (дашборд, сайт, пуши) — **вне этого репо** или отдельными пакетами; здесь — контракт и клиент.

---

## События и публичный API SDK (MVP → v1)

### Автоматически (ядро)

- Старт/конец процесса и **сессии foreground**: аналог `app_open` / `app_close` или пара `session_start` / `session_end` с длительностью.
- **`app_foreground` / `app_background`** — отсечь «открыл и свернул».
- **`heartbeat`** каждые 15–30 с **только пока приложение активно** (жизненный цикл + опционально простая «активность»).
- **`screen_view`** (ручной или через обёртку над навигацией) — минимум для доказательства движения по приложению.

### Опционально (расширение, явный вызов разработчика)

- **`trackInteraction()`** — «было действие», без детализации UI.
- **`trackEvent(name, props?)`** — ключевые шаги сценария (кнопка, отправка формы). **Без** автоперехвата всех кликов, **без** Accessibility как требования MVP.

### Связка с тестом и тестером

- **`init(apiKey | publishableKey, options)`** — ключ теста/проекта из кабинета.
- **`startTestSession(sessionId)`** или **`bindTester(testerId, testSessionId)`** после парсинга deep link / Intent.
- **Deep link / Intent:** хост приложения передаёт `test_session_id`, `tester_id` (или короткий код → ручной `bindTester` как fallback).
- Без привязки: на бэкенде остаётся **`device_id`** + агрегаты; точное «кто из списка тестеров» — только при успешном binding.

### Технические поля (контракт)

- `device_id` (стабильный UUID в хранилище приложения), `sdk_version`, `os`, `app_version`, монотонные timestamps, опционально `event_id` для дедупа.
- **Не собирать:** email Google «тихо», содержимое полей ввода, пароли; кастомные `props` — только без PII по договорённости.

## Приватность и формулировки (Play / доверие)

- В Data safety и в документации: **session time, screen names (как задаст разработчик), опциональные именованные события** — для проверки участия в closed testing, не «скрытая аналитика поведения».
- Позиционирование для стора вашего **хост-приложения** (если есть): валидация участия в тесте, а не «увеличение активности ради метрик».

---

## Open source и граница с бэкендом

| Открыть | Закрыть на сервере |
|--------|-------------------|
| Исходники SDK, формат payload, примеры | Пороги валидности сессии, scoring, анти-абуз эвристики |
| Документация «что уходит по сети» | Секреты только для сервера (если появятся) |

Идея: **Open SDK → доверие; «интеллект» валидации — на бэкенде.**

## Защита эндпоинтов (реалистично)

Полностью запретить имитацию из Postman **нельзя**; цель — **`POST` ≠ валидная сессия** для отчётов.

MVP на бэкенде (рядом с SDK в общей архитектуре):

1. **Handshake после `init`:** короткоживущий `session_token`, рефреш, привязка к `device_id` + ключу теста.
2. **Проверка времени:** heartbeat с реалистичными интервалами, лимиты частоты, запрет «свалки» событий за секунду.
3. **Порядок и типы событий:** допустимые переходы (например, нельзя бесконечно `session_end` без `session_start`).
4. **Rate limiting** по ключу и устройству.
5. Позже: **Play Integrity** / attestation по мере необходимости (усложняет интеграцию).

Подпись запросов секретом из **открытого** SDK даёт мало; опираться на **поведенческую валидацию** и токены сессии.

## Фазы работ (этот репозиторий + контракт с бэкендом)

### Фаза 0 — Спецификация (1–3 дня)

- Зафиксировать JSON-схему событий и имена (`session_start`, `heartbeat`, …).
- OpenAPI или аналог для **ingest** (init, batch events, refresh token).
- Политика PII и список запрещённых полей в `trackEvent`.

**Выход:** `SPEC.md` (или `docs/ingest.md`) в репо.

### Фаза 1 — Android SDK MVP (1–2 недели)

- AAR/модуль: `init`, lifecycle-сессии, heartbeat, опционально `screen`, `flush`.
- Персистентный `device_id`, очередь, батчи, ретраи, отключение в **test-only** / по флагу с сервера (вместо «удали SDK из кода», если решите так).
- Парсинг deep link / `bindTester` fallback.

**Выход:** опубликованный артефакт + sample app в `examples/`.

### Фаза 2 — Надёжность и DX

- Backoff, лимиты очереди, корректность при kill процесса.
- ProGuard/R8 rules, миниимальный размер.
- Версионирование SemVer, changelog.

### Фаза 3 — Расширение API

- `trackInteraction` / `trackEvent` с redaction/sanitize на клиенте.
- Опциональная интеграция с навигацией (Fragment/Compose) — отдельными модулями или докой «куда вставить один вызов».

### Фаза 4 — Усиление доверия к данным (по необходимости)

- Опционально Play Integrity, дополнительные серверные проверки.
- Клиентская эвристика `low_quality_hint` **только как подсказка**, финальный score — на сервере.

## Риски (кратко)

| Риск | Митигация |
|------|-----------|
| SDK воспринимают как «ещё одну аналитику» | Документация и нейминг под **closed testing / proof of usage** |
| Слабая привязка tester ↔ устройство без companion app | Deep link + код; в UI дашборда явно «неизвестный тестер / устройство N» |
| Модерация Play для вашего приложения-трекера | Не обещать обход правил Google; прозрачный сбор данных |
| Фейковые запросы к API | Токены, тайминг, лимиты; валидность сессии только после серверных правил |
